# 项目立项与迭代总结

## 项目目标

提升 `tgsearch115` 在 MoviePilot v2 中的资源匹配准确率和观影资源详情获取成功率，避免错误资源被自动转存并错误完成订阅。

## 核心指标

| 指标 | 当前基线 | 第一目标 |
|---|---:|---:|
| TG Top-1 正确率 | 约 20%（用户反馈） | >= 90% |
| 错误资源自动转存率 | 未量化 | <= 1% |
| 观影详情获取成功率 | 可搜作品但常无详情 | >= 90%（可访问样本） |
| 失败原因可诊断率 | 低，常折叠为空结果 | 100% 分类记录 |

### 根因一：TG 结果缺少媒体身份校验

当前链路是“片名召回 -> MP 质量规则过滤 -> 第一条 115 资源”。Telegram 服务端搜索本身是模糊召回，`tg_scraper.py` 明确不做本地关键词强匹配；而 `_filter_resources` 只处理资源偏好，不校验片名、年份、别名、TMDB ID、媒体类型或季信息。最终 `matched_115[0]` 使频道顺序和消息顺序意外成为选片依据。

此外，观影 `_get_items` 只有在找到同年份作品时才替换结果；找不到精确年份时会保留所有模糊作品，这不适合自动订阅流程。

### 根因二：观影详情接口失败被折叠为空结果

本地 `site_scraper.py` 对 `downurl` 非 200 响应只记录一条日志并返回 `[]`，没有重试、页面兜底或结构化错误。上游 `v4.2.17-v4.2.18` 已加入：

- 403/404 时携带 `Referer`、`Origin` 等请求头重试；
- API 受限时从页面提取网盘链接；
- 优先兜底提取磁力链接；
- 更彻底地隔离环境代理。

因此本地问题至少包含一个已确认的版本缺口。是否还存在 MoviePilot 宿主 IP 封锁、凭证失效或站点接口变更，需要运行时日志进一步确认。

## 阶段二：优化方案设计

日期：2026-07-19

状态：完成。

### P0：观影止血与可诊断化

1. 以独立提交同步上游 `v4.2.15-v4.2.18`，重点审查 `site_scraper.py`，不直接覆盖本地其它定制。
2. 将 `_fetch_resources` 从“只返回列表”改为返回结构化结果：`hits`、`status`、`http_status`、`error_code`、`detail`、`elapsed_ms`。
3. 详情获取按以下顺序执行：JSON 请求、带 `Origin/Referer` 重试、明确的 HTML 详情页兜底、聚影官方 API 兜底。每层只重试一次，禁止无界重试。
4. `check_site` 不再只测 `search_suggest`，必须选择一个搜索结果继续验证 `downurl`，分别报告“搜索可用”和“详情可用”。
5. PoW 只有服务端明确返回成功时才设置 `_pow_solved=True`；认证失效、WAF/IP 拒绝、接口变更、空资源必须使用不同错误码。

验收：准备至少 20 个可确认有资源的作品，详情成功率达到 90%；失败样本均能归类，不再统一显示“找到 0 条”。

### P1：TG 媒体身份硬过滤

在 `_filter_resources` 之前新增统一的 `MediaCandidate` 标准化层，输入订阅的 MoviePilot `MediaInfo` 和每条 TG/观影结果，输出标准化标题、别名、年份、媒体类型、季、集数与来源。

自动订阅硬条件：

- 标准化标题必须与主标题或别名精确相等，或满足边界清晰的完整标题匹配；仅有单字/短词包含不算命中。
- 候选明确带年份时必须与订阅年份一致；年份冲突直接淘汰。订阅流程中观影找不到精确年份时返回空结果，不再回退全部模糊作品。
- 电影候选不得带明显剧集季集标记；剧集候选的季号必须匹配目标季。
- TG 消息中的多个 115 链接分别绑定其附近文本，避免一条合集消息把所有链接错误套用同一标题。

验收：用至少 30 个真实查询建立固定样本，覆盖同名电影、翻拍年份、电影/剧集同名、季号、中文/英文/别名和合集消息；错误候选硬过滤召回率达到 100%，正确候选保留率不低于 95%。

### P2：可解释评分、去重与选择

硬过滤通过后再评分，初始建议权重如下，最终以样本集调参：

| 特征 | 建议分值 |
|---|---:|
| 标准化主标题精确匹配 | +50 |
| 标准化别名精确匹配 | +45 |
| 年份一致 | +20 |
| 媒体类型一致 | +10 |
| 季号一致 | +10 |
| 完结/集数覆盖订阅缺失范围 | +10 |
| 首选清晰度、编码、字幕 | +5 至 +15 |
| 年份冲突 | 直接淘汰 |
| 标题冲突或季号冲突 | 直接淘汰 |

按标准化分享 URL 或 `share_code` 去重；不再使用 `matched_115[0]`，改为按“身份分、订阅覆盖度、质量分、来源可靠度、发布时间”稳定排序。建议自动转存阈值为 80 分；60-79 分只展示并提示人工确认；低于 60 分丢弃。阈值需通过样本校准，不应直接凭经验上线。

### P3：影子运行与灰度发布

1. 新增 `dry_run`：执行搜索、过滤和排序，只记录最终候选及各项得分，不转存、不完成订阅。
2. 先运行 3-7 天，人工标注 Top-3 是否正确，计算 Top-1 准确率、正确资源召回率和各来源详情成功率。
3. 达标后只对高置信度电影开启自动转存，再逐步开放剧集；任一错误转存立即关闭自动模式并保留回退。
4. 每次发布保留上一个插件版本和配置导出，确保可以快速回滚。

### 日志与隐私

每次搜索生成 `trace_id`，记录查询、候选数量、淘汰原因、评分明细、最终选择和详情错误分类。日志必须脱敏：115 Cookie、`app_auth`、API Key、分享提取码和完整鉴权头不得输出。工作区诊断脚本中存在明文 `app_auth`，应立即作废该值并改用环境变量读取。

## 阶段三：MoviePilot 原生识别方案核对

日期：2026-07-19

状态：完成。

### 源码核对结论

已检查 `jxxghp/MoviePilot`、`jxxghp/MoviePilot-Plugins`，并专门核对当前运行版本对应的 `v2.13.16` 标签。以下接口在该版本存在：

- `app.core.metainfo.MetaInfo(title, subtitle)`：解析资源标题、年份、类型、季和集数；
- `app.helper.torrent.TorrentHelper.match_torrent(...)`：比较目标媒体的标题、原标题、别名、年份和类型；
- `app.chain.media.MediaChain.recognize_by_meta(...)`：通过 MoviePilot 当前配置的媒体识别模块获得候选 `MediaInfo`；
- `MediaInfo.tmdb_id`、`douban_id`、`names`、`season_years`：可用于强身份比对和别名处理。

MoviePilot 自身的订阅链路也是先识别候选，再比较媒体 ID；识别失败时才使用 `TorrentHelper.match_torrent` 做标题兜底。当前插件只调用 `SubscribeChain.filter_torrents`，该方法解决的是质量偏好过滤，不是媒体身份确认。

### 推荐接入方式

在插件 `_handle_subscribe` 中，将当前的：

```text
TG/观影结果 -> _build_torrents -> 质量过滤 -> matched_115[0]
```

调整为：

```text
TG/观影结果
  -> MetaInfo 解析候选标题和描述
  -> TorrentHelper.match_torrent 本地初筛
  -> 去重并选出 Top 3
  -> MediaChain.recognize_by_meta 确认候选身份
  -> 候选 TMDB ID == 订阅 TMDB ID
  -> 季号/缺集范围检查
  -> MoviePilot 质量规则过滤
  -> 评分最高者自动转存
```

建议新增 `_confirm_candidate_identity(subscribe, target_media, torrent)`，返回结构化结果：

```text
confirmed: bool
match_source: tmdb_id | douban_id | title_fallback | rejected
candidate_media_id: string
reason: string
```

确认规则：

1. 候选能够识别出 TMDB ID时，必须与订阅 TMDB ID 相等；不相等立即淘汰。
2. 订阅没有 TMDB ID但有豆瓣 ID时，比较豆瓣 ID。
3. 候选识别失败时，标题兜底结果不得直接自动转存，只能进入人工确认或回退 MoviePilot 默认搜索。
4. 剧集除 ID 相等外，还必须验证目标季和缺失集范围。
5. 观影 `source_title + year` 应作为识别标题，原始 `panlist.name` 只用于解析质量和集数。

### 性能与稳定性

不应对全部 TG 搜索结果逐条调用 TMDB。先用 `MetaInfo` 和 `TorrentHelper.match_torrent` 本地过滤、URL 去重和排序，只对 Top 3 调用 `recognize_by_meta`。按“标准化标题 + 年份 + 类型 + 季”缓存确认结果，建议缓存 24 小时。

不建议直接调用 TMDB HTTP API，因为这会绕过 MoviePilot 的识别缓存、模块切换、辅助识别、共享识别和限流处理。不建议直接调用 `SubscribeChain.match`，因为它面向全局订阅匹配流程，包含状态处理和下载相关逻辑，不适合作为插件内的纯校验函数。

### 修订后的实现优先级

1. P0：同步观影 `v4.2.18` 修复并补齐详情诊断。
2. P1：实现 `MetaInfo + TorrentHelper` 本地身份初筛，立即停止首条直转。
3. P2：为 Top 3 增加 `MediaChain.recognize_by_meta`，以 TMDB ID 作为自动转存硬门槛。
4. P3：影子运行验证准确率后再开启自动转存。

## 阶段十二 A：115 直接磁力接口可行性核查

日期：2026-07-20

状态：源码核查完成；宿主 Cookie 只读探测待 v4.7.0 客户端接入后执行。

### 接口结论

1. 现有 `p115_transfer.py` 的 `share_snap/share_receive` 是 115 分享转存接口，不支持磁力，不能用于直接离线下载。
2. 115 官方 Open API 提供 `add_task_urls`、`get_task_list`、`del_task` 等 `/open/offline/*` 接口，但依赖 OAuth 凭据；当前插件只有 Cookie，不把 Open API 当作默认实现。
3. 115 Cookie Web 云下载接口具备完整能力：`clouddownload&ac=space` 获取提交签名，`lixianssp/?ac=add_task_url` 创建磁力任务，`task_lists`/`get_user_task` 查询任务，`task_del` 取消任务，`restart` 重试任务。
4. 任务以 `info_hash` 为稳定标识，可取得任务状态、进度和错误字段。已核对客户端状态映射为失败、停止、运行和成功。
5. 115 返回成功只能证明离线任务完成，不能证明 MoviePilot 已整理入库；任务必须先进入 `pending_organize`，只有 MoviePilot 订阅完成历史才能确认最终 `completed`。

### 安全边界与验证计划

- 当前 v4.6.0 没有安全的云下载探测端点，不能在不把 Cookie 带出宿主的前提下完成实机签名与列表探测。

- 所有开发测试使用模拟响应，不提交真实磁力，不创建、删除、暂停或重试真实任务，不修改 115 目录。
- 若宿主只读探测最终表明 Cookie Web 接口不可用，默认 `direct_then_cms` 将安全回退 CMS，并在文档中标记直接接口不可用原因。
