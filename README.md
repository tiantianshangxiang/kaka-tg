# 拦截 MP 订阅（tgsearch115）

MoviePilot v2 插件：订阅新增或周期扫描时优先搜索 Telegram 公开频道、观影资源站和聚影 API；匹配的 115 分享直接转存，确认匹配的完整观影磁力通过 CMS 离线到 115，失败时回退到 MoviePilot 默认搜索。

当前本地版本：`v4.6.0`。该版本增加周期搜索、来源风控和 CMS 任务账本；只有 MoviePilot 出现订阅完成历史时才确认整理完成，超时会恢复订阅继续搜索。

## 目录结构

```text
├── package.v2.json
└── plugins.v2/tgsearch115/
    ├── __init__.py       # 订阅事件、匹配、排序、API
    ├── tg_scraper.py     # Telegram 公开频道网页搜索
    ├── site_scraper.py   # 观影 PoW、搜索和资源详情抓取
    ├── juying_scraper.py # 聚影开发者 API
    ├── cms_client.py     # CMS Token API / 115 磁力离线
    ├── p115_transfer.py  # 115 分享转存
    └── frontend/         # MoviePilot Vue 前端
```

## 初始问题基线（v4.2.14，已修复）

### 1. TG 搜索与订阅误匹配

- `tg_scraper.py` 使用 `t.me/s/{channel}?q={片名}` 获取服务端模糊搜索结果，但本地不再校验标题。
- `_filter_resources` 只应用 MoviePilot 的规则组及 include/exclude、清晰度等条件，不负责媒体身份校验。
- 订阅流程从过滤后的 115 结果中直接选择第一条，缺少片名、别名、年份、季集和来源可信度评分。
- 观影的年份过滤在“无精确年份结果”时会保留全部模糊结果，也可能把错误作品带入订阅链路。

因此当前“10 条约 2 条正确”符合代码行为：召回量提高了，但准确率没有对应的二次校验和排序保障。

### 2. 观影能搜到作品但取不到资源

- 本地 `v4.2.14` 的 `/res/downurl/{dir}/{id}` 遇到 403/404 会直接返回空数组，前端只能看到作品搜索成功，无法看到具体资源。
- 上游 `v4.2.17-v4.2.18` 已增加完整浏览器请求头重试、页面 HTML 兜底提取和磁力优先提取；本地尚未包含这些改动。
- MoviePilot 运行时仍需用结构化日志区分 `HTTP 403`、非 JSON、业务 `code != 200`、登录过期及“确实无资源”。

## 优化方向

1. 先同步并审查上游 `v4.2.18` 的观影修复，解决已知版本缺口。
2. 在 TG 和观影召回之后增加统一的媒体身份标准化、硬过滤和置信度评分。
3. 订阅自动转存设置置信度门槛；低置信度结果只展示，不自动完成订阅。
4. 为搜索、详情获取和选择结果增加可观测日志与回归样本集。

媒体确认应优先复用 MoviePilot `v2.13.16` 已有能力：

- `MetaInfo(title, subtitle)` 解析候选标题、年份和季集；
- `TorrentHelper.match_torrent(...)` 做低成本标题、别名、年份和类型初筛；
- `MediaChain().recognize_by_meta(...)` 通过 MoviePilot 配置的识别源确认候选身份；
- 候选 `tmdb_id` 与订阅 `MediaInfo.tmdb_id` 相同后，才允许进入自动转存。

不建议插件直接请求 TMDB API，也不建议直接调用 `SubscribeChain.match`：前者会绕过 MoviePilot 的缓存、识别源和限流策略，后者是全局订阅匹配流程，可能产生下载等额外副作用。

建议实施顺序：

```text
P0 观影止血与诊断
  -> P1 TG 身份硬过滤
  -> P2 统一评分、去重和自动转存阈值
  -> P3 影子运行、灰度启用和指标验收
```

自动订阅应遵循“宁可回退，不可错转”：标题或年份冲突直接淘汰；信息不足但可能正确的结果只进入手动搜索，不自动转存。

详细方案、阶段记录和验收指标见 [项目立项与迭代总结](./项目立项与迭代总结.md)。

### 2026-07-19 阶段三：MoviePilot 原生识别方案核对

### 2026-07-19 阶段七：观影 app_auth Cookie 规范化

- `v4.3.1` 在 MoviePilot 实机中已正确进入 urllib 降级，但详情仍返回 `urllib HTTP 403`。
- 对比配置和成功探针后确认，用户保存的是 `app_auth=值` 完整 Cookie 片段，插件却把整段作为 Cookie 值，导致请求实际登录态无效。
- 新增输入规范化：同时接受纯 token、`app_auth=token` 和 `Cookie: ...; app_auth=token; ...`。
- Windows 直连对照验证：规范化前 `downurl` 为 403，规范化后为 HTTP 200，并取得 33 条网盘链接。
- 插件版本升级为 `v4.3.2`，新增两项 Cookie 格式回归测试。
- 已部署到 MoviePilot `v2.13.16`：观影搜索“双姝美探（2026）”返回 37 条唯一详情资源，URL 全部非空且无警告。
- 部署后回归 TG“法警小队（2026）”：返回 12 条唯一链接、仅 1 个规范化标题，错误作品为 0。

### 2026-07-19 阶段八：聚影非开发者鉴权核对

- 聚影没有单独的“非开发者端口”；指南中的非开发者调用仍使用 `/api/dev/*`。
- 所有 7 个开发者端点均要求同时提供已审核的开发者 `AppID` 和调用用户自己的个人 `API Key`。
- “非开发者共享 10 次/分钟”表示非开发者通过他人的已审核 `AppID` 共享调用，并不表示个人 `API Key` 可以单独鉴权。
- 当前 MoviePilot 的聚影开关、域名、AppID 和 API Key 均为空，未写入或输出用户提供的密钥。
- 现有插件已经支持共享调用模式；取得可用的开发者 AppID 和聚影 HTTPS 域名后，只需在聚影配置页填入三项凭证并测试连通。

### 2026-07-19 阶段九：观影磁力优先接入 MoviePilot（后端）

- 自动订阅候选继续先经过 MoviePilot 规则组、include/exclude、画质参数和 TMDB/豆瓣身份确认。
- 启用观影磁力优先时，完整观影磁力排在 115 分享前；按 BTIH 去重，失败后仍可回退 115 或 MoviePilot 默认搜索。
- `v4.4.0` 曾把磁力作为 `TorrentInfo.enclosure` 提交给 MoviePilot `DownloadChain`；该路径已在 `v4.5.0` 被 CMS/115 官方 Token API 取代，当前磁力不会进入任意 MoviePilot 下载器。
- 剧集仅允许明确完结/全集磁力自动提交；未完结剧集、聚影磁力和其它来源磁力不参与该优先路径。
- 新增独立资源策略模块和 3 项回归测试；前端配置开关与手动搜索筛选控件将在同版本下一阶段完成。

- 配置弹窗和插件详情页新增一致的资源筛选按钮（全部/磁力/网盘/115）与画质按钮（全部画质/4K/1080P/高帧率/排除 HDR）。
- 前端筛选逻辑共用单一模块，已通过资源类型、画质和 HDR 排除断言；生产构建成功。
- 发布版本升级为 `v4.4.0`，完整测试扩展到 20 项并全部通过。

### 2026-07-19 阶段十：观影磁力固定通过 CMS 离线到 115

- 根据 `imaliang/cms-docs` 官方文档接入 `POST /api/cloud/add_share_down_by_token`，不再把观影磁力交给 MoviePilot 中任意下载器。
- 自动磁力在提交前仍依次经过 MoviePilot 规则组、include/exclude/画质过滤、中文片名与年份初筛、TMDB/豆瓣 ID 和剧集完整性确认。
- CMS 地址与 Token 只保存在 MoviePilot 插件配置；CMS 登录账号和密码不参与 API 调用，密钥不写入代码、README、日志或 Git。
- CMS 失败时跳过剩余磁力并继续尝试后续 115 分享；CMS 接口只表示“任务已创建”，插件仅暂停订阅等待同步，不再误发下载完成事件。
- 配置弹窗和插件详情页的磁力结果新增“离线到115”操作；高帧率筛选补充 `60帧`、`60帧率` 等中文形式。
- 本地已通过 23 个 Python 测试、Python 编译、Vue/Vite 生产构建、Node 筛选断言、清单解析和 `git diff --check`；未提交真实磁力任务。

### 2026-07-20 阶段十一 A：周期搜索与来源风控后端

- 新增单一有界优先队列；新增订阅、周期扫描和手动搜索不再各自无限创建线程，手动搜索在等待队列中优先于周期任务。
- 周期扫描只处理 MoviePilot `state=N` 的活动订阅，并按片名、年份、类型和季去重；默认每 2 小时运行，每轮增加 0-10 分钟随机抖动。
- 自动搜索增加按来源 TTL 缓存；TG 默认并发降为 2，翻页间隔为 0.8-1.5 秒；观影详情间隔为 1.5-3 秒。
- TG 429、观影 403/429 支持退避并读取 `Retry-After`；连续失败达到阈值后来源进入 30-60 分钟冷却。
- CMS 任务账本持久化订阅 ID、媒体 ID、BTIH、标题、时间、状态和错误；跨重启按 BTIH 去重。
- CMS 接收任务后不会完成订阅；只有 MoviePilot 订阅历史出现或订阅被 MP 整理流程移除时才标记完成，超时则恢复订阅为 `state=N`。
- 34 个 Python 测试通过；下一步补齐配置界面、状态页和宿主验证后再发布 `v4.6.0`。
